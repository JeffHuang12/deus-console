# 第一部分：三項串接的可行性與工時量級

工時量級採級距，非精確估算：**小時級 / 天級 / 週級 / 月級**。所有牽涉外部審核（如 Meta App Review）的項目，工時受第三方審核時間支配，標 [推論]。

## 1. OAuth / 資料源綁定

### 現況

- 接點：`backend/integrations/oauth.py`，目前 `start_flow()` 回 stub、`verify_shopline_token()` 一律回 True。
- 前端：`/binding` 頁十張資料源卡片 UI 完成，連線為 stub。
- GA4 例外：已用 ADC（`ga4/ga4_adc.json` authorized_user）打通，`ga4_agent.py` 實跑。

### 行動與工時（逐源）

| 資料源 | 綁定方式 | 憑證是否就緒 | 要做的事 | 可行性 | 工時量級 |
|--------|----------|--------------|----------|--------|----------|
| GA4 | OAuth/ADC | 已就緒 | 已完成，沿用 `ga4_agent` 模式 | 已完成 | 0 |
| Shopline | Access Token | 已就緒 | 實作 `verify_shopline_token`（呼 Open API token info），沿用 `shopline_getorders/shopline_common.py` | 高 | 小時級 |
| Boxful | API Key | 已就緒 | Modal 收 key + 後端驗證一次呼叫 | 高 | 小時級 |
| Google Ads | Google OAuth | 已就緒（refresh token 在 .env） | 用既有 refresh token 初始化 client，無需重走授權 | 高 | 天級 |
| Search Console | Google OAuth | 部分（可共用 Google client） | 加 scope、走一次授權、存 token | 高 | 天級 |
| Meta 廣告 | Meta OAuth | 已就緒（access token 在 .env） | 用既有 token 接 Graph API；風險為 token 到期與權限範圍 | 中高 | 天級～週級 |
| Meta 粉專 / IG / Messenger | Meta OAuth | 部分 | 需擴充權限範圍，部分 scope 須 App Review | 中 | 週級～月級 [推論，審核主導] |
| Threads | Threads API | 未就緒 | 獨立 API 與授權，須申請 | 中 | 週級～月級 [推論] |
| LINE LAP / OA | LINE 授權 | 未就緒 | 需 LINE Business + channel 與 token | 中 | 週級 |
| Bing Ads | Microsoft OAuth | 未就緒 | 註冊應用、走 MS OAuth | 中 | 天級～週級 |

### 結論

- 工程上 OAuth 不難，難在第三方審核與權限範圍。建議分批接通，依憑證就緒度排序：**GA4（已完成）→ Shopline / Boxful → Google Ads / Search Console → Meta 廣告 → 其餘需審核者**。
- 統一介面已在 `oauth.py` 預留，逐源實作 adapter，router 不直接耦合。

## 2. BigQuery 查詢層

### 現況

- 接點：`backend/integrations/bigquery_client.py`，`query()` 目前直接 `raise NotImplementedError`。
- 基礎設施已備：`.env` 已有 `GOOGLE_CLOUD_PROJECT`、`BIGQUERY_DATASET`、`GOOGLE_APPLICATION_CREDENTIALS`。
- 既有資產：Shopline → BigQuery 的每日 ETL 已在運作（VITABOX DEUS 專案），代表 dataset、schema、權限鏈路已存在。

### 行動與工時

| 工作項 | 內容 | 可行性 | 工時量級 |
|--------|------|--------|----------|
| 實作 `query()` | 用 `google-cloud-bigquery`，`bigquery.Client()` 從 .env 讀設定，回 `[dict(row) ...]` | 高 | 小時級 |
| 分析中心改讀真資料 | 八區塊的規則計算改打 BigQuery SQL，取代 `mock_data`；含日期區間、資料源、對比期間參數化 | 高 | 週級（取決於報表數與 schema 熟悉度） |
| 監測中心排程 | 排程器定時跑 BQ 比對門檻後告警 | 中高 | 天級～週級 |
| 成本與延遲控管 | 查詢分區、限制範圍、預先彙整、快取 | 中 | 天級（持續優化） |

### 結論

- 可行性高，這是三項裡風險最低的。`query()` 本身是小時級工作。真正的工是「知道 BQ 裡有哪些表與欄位、把分析邏輯寫成 SQL、把各 router 從 mock 切到 BQ」，量級落在週級。
- 風險為查詢成本與延遲，用分區/快取/預彙整因應（prd §7.2 已列）。

## 3. Claude API

### 現況

- 接點：`backend/integrations/llm.py`，`refine_insight()` 與 `chat()` 皆回 stub 文字。
- `.env` 無 `ANTHROPIC_API_KEY`。
- 但 `ga4_agent.py` 已證明可用 Claude Agent SDK + Claude Code CLI 訂閱驅動 Claude（不走 API key 計費）。

### 兩條落地路徑

- **路徑 A：anthropic SDK + API key（按 token 計費）**。需在 .env 補 `ANTHROPIC_API_KEY`。適合正式部署、可控併發與計費，支援 prompt caching。
- **路徑 B：沿用 CLI 訂閱模式（如 ga4_agent）**。不需 API key、不額外計費，但綁定本機 CLI 與訂閱，較適合本機/內部工具，不適合多人正式服務。

### 行動與工時

| 工作項 | 內容 | 可行性 | 工時量級 |
|--------|------|--------|----------|
| 決定路徑 A 或 B | 依是否正式對外、計費容忍度決定 | — | 小時級（決策） |
| `refine_insight()` | 把規則計算結果餵 prompt，回潤飾後段落；開 prompt caching | 高 | 天級 |
| `chat()`（/prompt 頁） | 帶 history、系統提示，必要時掛 MCP 工具 | 高 | 天級 |
| 成本與品質控管 | 混合策略（規則算數字 + LLM 潤飾文字）、模型可選、caching | 中 | 天級（持續調整） |

### 結論

- 可行性高。混合策略已在設計中（數字由程式掌握、敘述由模型掌握），與 `ga4_agent.py` 既有做法一致，可直接沿用避免幻覺。
- 先決策路徑 A/B。若要正式對外，走 A 並補 `ANTHROPIC_API_KEY`；若先內部驗證，走 B 最快。

## 4. 建議落地順序（總結）

1. **BigQuery `query()`**（小時級）→ 解鎖分析/監測的真實數據基礎。
2. **Claude API `refine_insight` / `chat`**（天級）→ 分析潤飾與 Prompt 互動立即可用。
3. **Shopline / Boxful / Google Ads 綁定**（小時～天級，憑證已備）。
4. **分析中心切真資料**（週級）→ 達成 prd 的 M3 里程碑。
5. **Meta / 其餘需審核資料源**（週～月級，並行送審）。

## 盤點總結

- **已在頁面（核心介面已具備）**：受眾文字描述→名單、廣告報表自然語言摘要、LLM 自動回覆留言、會員分群上傳廣告平台（Custom Audience）、GA4 自然語言查詢（唯一真接通）。
- **部分對應（有頁面承接，邏輯未實作）**：RFM 分群、各類受眾建立與排除、訊息推播（棄單/回購/序列/喚回）、留言標籤分類、監測規則、Prompt 互動（會員自然語言查詢的容器）。這些多半是「UI 框架已在、待接 BigQuery 取數＋Claude 生成＋平台回寫」。
- **未實作（Phase 2 之後路線）**：絕大多數預測型與生成型 AI 功能（CLV/流失/回購預測、Lookalike、RAG 客服 Bot、個人化內容生成、商品目錄優化、多通路歸因等），尤其 PDF 標 [中]/[高] 門檻者。

現有八頁無法完整承接 PDF 規劃，建議 Phase 2 後評估新增（依 HANDOFF 第 5 節九步流程實作，httpApi 與 mockApi 並行）：
